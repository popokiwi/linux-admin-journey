# Часть 4. Внутренний путь TCP и HTTP запроса

## Почему я начала изучать эту тему

Я уже знала общую цепочку:

браузер -> HTTP -> nginx -> Gunicorn -> Flask

Но мне было непонятно, что происходит между соседними компонентами.

Я знала назначение терминов TCP, HTTP и nginx, но не видела механизма, который связывает их в одну систему.

# TCP-соединение

Я начала разбирать TCP-handshake:

- SYN
- SYN-ACK
- ACK

Я поняла общую последовательность:

браузер просит операционную систему установить TCP соединение;
ядро выполняет TCP handshake;
после успешного connect() соединение установлено;
только после этого браузер может отправлять HTTP-запрос.

# HTTP и передача байтов

Я поняла, что HTTP-запрос сначала формируется браузером, а затем представляется в виде байтов.

Браузер вызывает:

send()

Но send() не передаёт запрос напрямую nginx.

Сначала байты попадают в отправляющий буфер ядра на стороне клиента.

Главный вывод:


`Возврат из send() не означает, что nginx уже получил или обработал запрос.
`

Общая причинная модель

Я начала связывать между собой:

- процесс браузера;
- системные вызовы;
- ядро;
- сокет;
- TCP;
- HTTP;
- буферы;
- nginx;
- Gunicorn;
- Flask;
возвращаемые значения функций.

Функции вызывают другие функции, передают управление, получают результаты и продолжают выполнение. Поэтому общая система содержит переходы вперёд и возвраты назад

# Результат

TCP, HTTP, nginx, Gunicorn и Flask начали складываться для меня в единую причинно следственную систему.

Я перестала воспринимать их как отдельные несвязанные термины и начала понимать механизм перехода между уровнями.
