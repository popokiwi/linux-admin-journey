
# Часть 2. Service user, права Linux и первые инциденты

## Проблема запуска от root

Первоначально приложение работало с избыточными привилегиями.

Я разобралась, почему веб приложение не должно запускаться от `root`:

- `root` имеет полный контроль над системой;
- приложение работает с правами своего процесса;
- при компрометации приложения атакующий может получить права этого процесса;
- отдельный ограниченный пользователь уменьшает возможный ущерб;
- приложению не нужны права на всю операционную систему.

## Создание service user

Я создала отдельного системного пользователя для FlightSim:

```text
mrpopo
```
Его параметры:
```text
shell: /usr/sbin/nologin
home: /opt/flightsim
group: mrpopo
sudo: отсутствует
```
Я проверяла пользователя командами:

getent passwd mrpopo
getent group mrpopo
id mrpopo
groups mrpopo

Я поняла различие между обычным пользователем и service user:

- mrpopo не предназначен для обычного входа в систему;
- ему не нужна интерактивная оболочка;
- ему не нужен sudo;
- он должен иметь только права, необходимые Gunicorn.

После этого Gunicorn перестал запускаться от root.

## Изучение прав Linux

Файлы проекта остались во владении:

`root:root`

Пользователь mrpopo может читать код приложения, но не может изменять app.py или создавать файлы в корне /opt/flightsim

Я разобрала:

- owner;
- group;
- others;
- права r, w, x;
- режимы 644 и 755;
- различия прав файла и директории.

Для файла:

- r позволяет читать содержимое;
- w позволяет изменять содержимое;
- x позволяет выполнять файл

Для директории:

- r позволяет увидеть список имён;
- x позволяет войти в директорию и обращаться к объектам;
- w позволяет создавать, удалять и переименовывать объекты.

Я научилась проверять доступ от имени service user:

 `sudo -u mrpopo`

 ## Знакомство с systemd unit

 Я разобрала назначение параметров:

- User=;
- Group=;
- WorkingDirectory=;
- ExecStart=;
- Restart=;
- RuntimeDirectory=;
- Environment=.

Я начала использовать:

- systemctl status;
- systemctl restart;
- systemctl daemon-reload;
- journalctl;
- systemd-analyze verify.

Я поняла, что:

- после изменения unit файла нужен daemon-reload;
- затем сервис необходимо перезапустить;
- журнал помогает найти причину неудачного запуска;
- предупреждения systemd-analyze verify могут относиться к другим сервисам.

## инцидент: runtime файл Gunicorn

После запуска Gunicorn от пользователя mrpopo в журнале появилась ошибка:

`Permission denied: '/opt/flightsim/.gunicorn'`

Я выяснила:

- домашней директорией mrpopo является /opt/flightsim;
- переменная XDG_RUNTIME_DIR не была задана;
- Gunicorn пытался создать служебный файл в /opt/flightsim/.gunicorn;
- /opt/flightsim принадлежит root:root;
- у mrpopo нет права записи.

Я поняла, почему не нужно:

- использовать chmod 777;
- передавать весь проект пользователю приложения;
- выдавать Gunicorn дополнительные привилегии.

Для runtime-файлов было создано отдельное место:

`RuntimeDirectory=flightsim
Environment="XDG_RUNTIME_DIR=/run/flightsim"`

Теперь systemd:

- создаёт /run/flightsim;
- назначает владельца mrpopo:mrpopo;
- передаёт путь Gunicorn;
- позволяет Gunicorn создавать там служебные файлы.

С помощью findmnt я проверила, что /run использует tmpfs.

Я поняла, что:

- runtime файлы являются временными;
- они должны храниться отдельно от кода;
- содержимое /run воспроизводится после перезапуска;
- systemd может автоматически создавать необходимые директории.

## Проверка разных участков системы

Я научилась проверять Gunicorn напрямую:

`curl -i http://127.0.0.1:8000/`

Этот запрос проверяет:

`curl -> Gunicorn -> Flask`

Проверка через nginx:

`curl -i http://127.0.0.1/`

проверяет:

`curl -> nginx -> Gunicorn -> Flask`

Я поняла, что разделение цепочки на участки позволяет быстрее определить место проблемы.

## Результат

Gunicorn работает от ограниченного пользователя mrpopo.

Код приложения остаётся защищённым, runtime-файлы создаются в отдельной директории, а я получила первый опыт полноценного расследования реальных проблем.
